# Product Proposal (PoC): Агент “Питание + готовка + календарь”

- Обоснование идеи – четкое описание прикладной задачи
- Цель и метрики – цель проекта и конкретные метрики успеха (продуктовые, агентские, технические)
- Потенциальные сценарии использования, в том числе с edge-кейсами
- Ограничения – предварительная оценка технических (SLO, p95 latency, …) и операционных (бюджет, …) ограничений
- Архитектурный набросок – краткое описание того, какие модули, интеграции нужны и будут задействованы для решения задачи
- Потенциальный data flow, какие части могут быть делегированы LLM/Agent, какие нет

## 1) Обоснование идеи

У большинства “планировщиков питания” есть две системные проблемы:
1. **План не исполняется**: рецепты выбираются без учета реальных временных слотов в календаре (учёба/работа/дорога), поэтому готовка “съезжает” и всё ломается к середине недели.
2. **Ограничения неполные/конфликтные**: диеты, аллергены, бюджет, остатки продуктов, доступность ингредиентов в доставке → требуют постоянного перепланирования.

**Задача PoC:** агентная система, которая строит **выполнимый недельный план питания**, привязывает готовку к **реальным** окнам в календаре, генерирует **список покупок** и умеет **перепланировать** при сбоях (нет ингредиента, не успел приготовить, изменился календарь), сохраняя историю и предпочтения.

## 2) Цель и метрики

### 2.1 Продуктовая цель

Улучшение опыта готовки и правильного питания дома: сделать так, чтобы пользователь следовал плану питания (минимум ручной рутины), при этом соблюдал ограничения (калории/БЖУ, аллергены, бюджет, время). 

### 2.2 Метрики успеха

**Продуктовые:**
- **Plan adherence:** доля приёмов пищи, отмеченных как “сделано/съедено” ≥ 60–70%
- **Calendar feasibility:** доля блюд, для которых есть достаточный слот в календаре (готовка + буфер) ≥ 90%
- **Constraint compliance:** доля планов без нарушений аллергенов/диет/явных запретов = 100% (строго)
- **Waste reduction proxy:** доля использованных продуктов из скоропортящихся ≥ X (например, 50%)

**Агентские (качество решений агента):**
- **Replan success rate:** доля “сбоев” (нет ингредиента/сорвалась готовка), после которых агент предлагает выполнимую альтернативу за ≤ 2 итерации ≥ 80%.
- **Recipe grounding score:** доля рекомендаций, обоснованных данными (нутриенты/ингредиенты/время готовки) и с корректными ссылками на источники ≥ 95%.
- **Hallucination rate:** доля фактов, не подтвержденных API/KB (калории, ингредиенты, времена) ≤ 2–5%

**Технические (SLO/наблюдаемость):**
- **Tool reliability:** успешность вызовов внешних API ≥ 95% (с ретраями и кэшем).
- **Privacy logging:** 0 случаев попадания PII/health-data в незащищённые логи (проверка/ревью).

## 3) Сценарии использования

### 3.1 Основной сценарий: “Сделай план на неделю”
1. Пользователь задаёт цель (калории/БЖУ/вес), ограничения (аллергии, “без рыбы”), бюджет, предпочтения, оборудование (духовка/мультиварка), число порций.
2. Агент читает **занятость календаря** и находит слоты для готовки (например, 30–45 мин вечером + 1 большой слот в выходные).
3. Агент выбирает рецепты, оптимизирует под текущие продукты (что нужно “спасти”), считает нутриенты, формирует расписание “что готовить когда”.
4. Генерирует список покупок, агрегированный по категориям

### 3.2 Сценарий: “Готовить некогда, но хочу правильно питаться”
- В календаре почти нет слотов → агент предлагает стратегию:
    - готовка “батчами” в один слот,
    - простые блюда ≤ 15 минут,
    - полуфабрикаты/готовая еда с контролем нутриентов

### 3.3 Сценарий: “Перепланирование по факту”
- Пользователь отмечает: “не успел приготовить ужин во вторник”.
- Агент:
    - переносит блюдо на ближайший возможный слот,
    - или заменяет на быстрое,
    - обновляет список покупок/остатков.

### 3.4 Edge-кейсы
- **Конфликт ограничений:** “фастфуд + 1000 калорий” → агент должен объяснить невозможность/нежелательность и предложить компромиссы (поднять калории/выбрать полезную альтернативу и т.п.).
- **Аллерген в “скрытом” ингредиенте:** соусы/полуфабрикаты, следы орехов → агент обязан маркировать риск, не утверждать безопасность без данных, предложить альтернативу.
- **Нет данных по нутриентам/ингредиентам в API:** fallback — другой источник/упрощённая оценка с пометкой “оценочно” или исключение рецепта.
- **Изменился календарь (встреча добавилась):** система должна пересчитать осуществимость и предложить перепланирование, не “молча” ломая план.
- **Недоступность продукта в доставке / отсутствует API магазина:** fallback — замены, или генерация “универсального списка покупок” без привязки к магазину.

## 4) Ограничения (технические и операционные)
### 4.1 Технические ограничения (предварительные)
- **SLO по задержкам**: недельный план допускает 1-2 минуты; интерактив ≤ 10.
- **Rate limits API:** минимизация запросов
- **Надёжность:**
    - ретраи,
    - кэширование результатов рецептов/нутриентов на сутки,
    - деградация функционала (если недоступны БД с рецептами/КБЖУ — предлагать рецепты без расчёта, но помечать и/или не использовать в “строгом режиме”).

### 4.2 Операционные ограничения
- **Бюджет:** не больше 5$ на пользователя в неделю
- **Доступность ЛЛМ:** запросы в API, невозможность поднять собственную


## 5) Архитектурный набросок (модули и интеграции)
### 5.1 Основные модули
1. **Orchestrator (агент/контроллер):** ведёт диалог/задачу, управляет шагами, решает когда звать инструменты, хранит state
2. **User Profile & Memory:**
    - долгосрочные предпочтения, запреты, цели,
    - история “понравилось/не понравилось”, “не готовлю > 30 мин”.
3. **Pantry / Inventory store:**
    - продукт, количество (примерно), срок годности (если есть/из таблицы)
4. **Recipe & Nutrition Retriever:**
    - поиск рецептов по ограничениям,
    - получение нутриентов, ингредиентов, времени готовки.
5. **Constraint Validator:**
    - аллергены/запреты (строго),
    - проверка калорий/БЖУ в пределах допусков,
    - проверка времени готовки vs календарные слоты,
6. **Scheduler (планировщик):**
    - выбирает слоты из календаря,
    - назначает “готовку” и “приём пищи”, добавляет буферы,
    - поддерживает перепланировку.
7. **Shopping List Builder:**
    - агрегирует ингредиенты с учетом продуктов, которые уже есть
    - предлагает замены,
    - экспорт: чеклист/CSV/заметки.
8. **Observability & Evals:**
    - лог причин отклонения рецептов/перепланирования,
    - метрики latency/cost/tool errors,
    - оффлайн-тесты на синтетических профилях.

### 5.2 Интеграции (для PoC минимум)
- **Calendar API:** Google Calendar (read free/busy, create events “Cooking: …”).
- **Recipes/Nutrition API:** Spoonacular или Edamam (или локальный датасет + простая нутри-база).
- **Grocery:** на PoC достаточно “генерация списка”

## 6) Потенциальный data flow 
### 6.1 Data flow
1. **Input:**
    - предпочтения/ограничения (UI/чат),
    - календарь (free/busy + фиксированные события),
    - продукты (ручной ввод / импорт),
    - цель по калориям.
2. **Normalize & Store:**
    - нормализация единиц, тегирование аллергенов, запись в state.
3. **Candidate generation:**
    - поиск 30–100 кандидатов рецептов через API/KB.
4. **Filter:**
    - исключить по аллергенам/оборудованию/времени, проверка на повторы, базовый бюджет-фильтр.
5. **Plan optimization (частично LLM + частично алгоритмически):**
    - подбор набора рецептов под неделю (разнообразие, повтор ингредиентов, цели по нутриентам),
    - выбор слотов в календаре, батч-готовка.
6. **Validation gatе:**
    - финальная проверка ограничений
7. **Output:**
    - недельное меню + расписание готовки + список покупок + “что из текущих продуктов будет использовано”,
    - опционально: события в календаре.
8. **Runtime monitoring:**
    - пользователь отмечает выполнение/пропуск → перепланирование.

### 6.2 Что можно делегировать LLM/агенту
- Интерпретация предпочтений в “структурные” ограничения (например, “хочу больше белка и без молочки” → JSON constraints).
- Генерация объяснений “почему такой план” и “почему этот рецепт подходит”.
- Предложение замен ингредиентов (с последующей проверкой правил/аллергенов).
- Выбор компромисса при конфликтующих ограничениях (но с детерминированной проверкой результата).

### 6.3 Что **нельзя** оставлять только на LLM (нужно правило/валидация)
- **Аллергены/строгие запреты**: проверка по структурам ингредиентов/тегам, не “на глаз”.
- **Расчёт нутриентов/калорий**: только из источника (API/KB) + явная пометка, если оценка приблизительная.
- **Проверка осуществимости по времени**: слот в календаре vs время готовки + буфер.
- **Формирование списка покупок с количествами**: единицы измерения/агрегация
- **Запись в календарь/изменения state**: только через “confirm/apply” (план изменений), чтобы избежать опасных действий
